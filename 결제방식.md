# 결제방식 변경 문서

## 목적

현재 워크숍 신청 결제 흐름은 Toss Payments SDK와 Toss 승인 API에 의존한다. 앞으로는 Toss를 사용하지 않는 방식으로 전환한다.

이 문서는 결제 담당 개발자가 PR을 올릴 때 지켜야 할 범위, 유지해야 할 DB 계약, 제거해야 할 Toss 의존성, 검증 기준을 정리한다.

## 결정 사항

- Toss Payments는 사용하지 않는다.
- `@tosspayments/payment-sdk`, `@tosspayments/payment-widget-sdk` 의존성을 제거한다.
- `NEXT_PUBLIC_TOSS_CLIENT_KEY`, `TOSS_SECRET_KEY` 환경변수는 더 이상 사용하지 않는다.
- 워크숍 신청 데이터의 진실의 원천은 Supabase의 `workshop_registrations_v2`와 `payments` 테이블이다.
- 신청 생성은 계속 `create_pending_registration(p_workshop_id)` RPC를 통해서만 처리한다.
- 결제 완료 후 신청 확정은 계속 서버에서만 처리한다. 클라이언트가 직접 `confirmed` 상태로 바꾸면 안 된다.

## 기본 전환 방향

새 결제 방식이 아직 최종 확정되지 않았다면, 1차 구현 기준은 **계좌이체 또는 외부 결제 확인형**으로 둔다.

사용자 흐름:

1. 사용자가 로그인하고 프로필을 완성한다.
2. 워크숍 상세에서 일정이 필요한 경우 일정을 선택한다.
3. 신청 버튼 클릭 시 `create_pending_registration(p_workshop_id)` RPC를 호출한다.
4. 서버가 `registration_id`, `order_id`, `amount`, `workshop_title`을 반환한다.
5. 프론트는 Toss 결제창 대신 결제 안내 화면 또는 모달을 보여준다.
6. 사용자는 안내된 방식으로 결제를 진행한다.
7. 결제 확인은 서버 라우트, 관리자 기능, 또는 새 결제사의 검증된 웹훅에서만 수행한다.
8. 확인이 끝나면 서버가 `confirm_payment_registration(...)` RPC를 호출해 `confirmed`로 전환한다.

새 PG사를 붙이는 경우에도 위 흐름은 동일하다. 단, 5번이 새 결제사 checkout 이동으로 바뀌고 7번이 새 결제사 승인 API 또는 웹훅 검증으로 바뀐다.

## 유지해야 할 DB 계약

기존 신청 계약은 유지한다.

- 테이블: `workshop_registrations_v2`
- 상태값:
  - `pending`: 신청 생성 후 결제 또는 확인 대기
  - `confirmed`: 결제 확인 완료
  - `cancelled`: 결제 실패, 사용자 취소, 관리자 취소
  - `expired`: 제한 시간 초과
- 결제 기록 테이블: `payments`
- 신청 생성 RPC: `create_pending_registration(p_workshop_id)`
- 결제 확정 RPC: `confirm_payment_registration(p_registration_id, p_payment_key, p_order_id, p_amount)`

`confirm_payment_registration`의 `p_payment_key`에는 새 방식의 고유 결제 식별자를 넣는다.

예시:

- 새 PG사 사용: PG사가 발급한 transaction id
- 계좌이체 수동 확인: `manual:{registration_id}:{approved_at}` 형식의 내부 승인 키
- 외부 결제 링크 사용: 외부 서비스의 payment id 또는 receipt id

`p_payment_key`는 `payments.payment_key`의 UNIQUE 제약을 만족해야 한다.

## 보안 원칙

- 클라이언트에서 결제 완료를 신뢰하지 않는다.
- 클라이언트에서 `workshop_registrations_v2` 또는 `payments`를 직접 insert/update 하지 않는다.
- 결제 확정은 서버 라우트, 검증된 웹훅, 또는 관리자 권한이 있는 서버 액션에서만 처리한다.
- 새 PG사를 쓰는 경우 웹훅 서명 검증을 반드시 구현한다.
- 계좌이체 수동 확인 방식이면 관리자 권한 검증이 필수다.
- 결제 금액은 클라이언트 요청값이 아니라 DB의 `registration.amount`를 기준으로 검증한다.
- `order_id`, `amount`, `registration_id`, `user_id` 일치 여부를 확인한 뒤 확정한다.
- 카드번호, 계좌 비밀번호, 주민등록번호 등 민감 결제 정보는 이 서비스에 저장하지 않는다.

## 새 API 설계 가이드

새 결제 방식에 따라 아래 중 하나로 구현한다.

### 1. 계좌이체 또는 수동 확인형

권장 API:

- `POST /api/payment/request`
  - 신청 생성 후 결제 안내 정보를 반환한다.
  - 반환 예시: `registration_id`, `order_id`, `amount`, `depositName`, `expiresAt`
- `POST /api/admin/payment/confirm`
  - 관리자만 호출 가능하다.
  - 입금 확인 후 `confirm_payment_registration(...)` RPC를 호출한다.
- `POST /api/admin/payment/cancel`
  - 관리자만 호출 가능하다.
  - 미입금, 오입금, 사용자 취소 등의 경우 `cancelled`로 전환한다.

### 2. 새 PG사 checkout형

권장 API:

- `POST /api/payment/checkout`
  - 서버에서 신청 정보를 검증하고 checkout URL 또는 결제 세션을 생성한다.
- `POST /api/payment/webhook`
  - PG사 웹훅을 받는다.
  - 웹훅 서명을 검증한다.
  - 결제 상태, 금액, 주문번호를 검증한다.
  - 검증 완료 후 `confirm_payment_registration(...)` RPC를 호출한다.
- `GET /payment/success`
  - 결제 결과를 직접 확정하지 않고, 신청 상태 조회 또는 안내 표시만 한다.
- `GET /payment/fail`
  - 실패 안내 표시 후 필요하면 서버 라우트로 취소 처리한다.

## 상태 처리 기준

- 신청 직후 상태는 항상 `pending`이다.
- 결제 확인 전에는 정원 계산에서 `pending`과 `confirmed`가 중복 신청 방지 대상이다.
- 결제 확인 완료 시에만 `confirmed`로 바꾼다.
- 결제가 실패하거나 사용자가 취소하면 `cancelled`로 바꾼다.
- 제한 시간 안에 결제가 확인되지 않으면 `expired`로 바꾼다.
- 새 방식에서 결제 시간이 15분보다 길어질 수 있으면 `expires_at` 정책을 함께 수정해야 한다.

## UI 요구사항

- 신청 버튼 문구는 결제 방식에 맞게 수정한다.
  - 예: `신청하기`, `입금 안내 받기`, `결제하기`
- 결제 안내에는 최소한 아래 정보를 표시한다.
  - 워크숍명
  - 결제 금액
  - 주문번호 또는 신청번호
  - 결제 기한
  - 확인 방식
  - 취소 및 환불 안내
- 사용자가 결제 완료를 눌러도 즉시 `confirmed`로 처리하지 않는다.
- 결제 확인 대기 상태를 명확히 보여준다.

## PR 완료 조건

PR은 아래 조건을 만족해야 한다.

- Toss SDK import가 남아 있지 않다.
- Toss 환경변수 사용이 남아 있지 않다.
- Toss API URL 호출이 남아 있지 않다.
- 신청 생성은 `create_pending_registration` RPC를 사용한다.
- 결제 확정은 서버에서만 수행한다.
- 결제 확정 시 `registration_id`, `order_id`, `amount`, `user_id` 검증이 있다.
- 결제 확정은 `confirm_payment_registration` RPC를 통해 처리한다.
- 실패 또는 취소 처리 경로가 있다.
- 중복 confirm 요청에도 안전하다.
- `npm run lint`가 통과한다.
- `npm run build`가 통과한다.

아래 항목은 구현 전에 최종 결정이 필요하다.

- 최종 결제 방식: 계좌이체, 외부 결제 링크, 새 PG사, 현장 결제 중 무엇인지
- 입금 계좌 또는 외부 결제 서비스 정보
